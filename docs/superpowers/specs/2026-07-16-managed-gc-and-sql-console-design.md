# 受控 GC 与本地 SQL 调试台设计

## 目标

为后台提供可扩展、可追踪的数据生命周期管理能力，并在本地开发环境提供受限的 SQL 调试入口。

## GC 资源模型

GC 不直接对任意表执行通用删除。每个可清理业务数据由所属模块实现 `GcResource` 并注册为 Spring Bean；资源定义清理条件、预览逻辑和分批删除逻辑。资源策略在 `maintenance_gc_policies` 中持久化，新增资源启动时自动创建默认策略，因此后台无需为每个表新增页面或接口。

首批资源为 Refresh Token 会话、审计日志、限流内存窗口、登录失败内存窗口。用户、角色、权限、菜单、公告、简历文档和简历反馈不会自动注册为可清理资源。

## 策略与执行

每个资源拥有独立的启用状态、自动执行状态、保留天数和批大小。保留期为 `0` 时禁用该资源的持久化数据清理。全局运行时设置仅控制 GC 总开关、调度间隔和单次最大持续时间。

GC 支持 dry-run、手动执行和自动执行。每次运行保存来源、资源范围、策略快照、扫描/删除数量、状态、错误摘要、Trace ID 和时间。数据库锁确保多实例下同一时刻仅有一个运行。任务按资源隔离：单一资源失败会记录失败但不会阻断其他资源。

## 权限与 API

- `system:gc:read`：查看资源、策略、预览与运行历史。
- `system:gc:write`：调整策略、运行、取消、暂停或恢复。
- `system:sql:execute`：仅本地超级管理员执行 SQL。

GC API 位于 `/api/v1/system/gc`，覆盖状态、资源策略、预览/运行、运行历史和取消请求。后台通过资源列表动态渲染策略表。

## 本地 SQL 调试台

SQL 调试台位于 `/api/v1/system/sql-console`。它只能同时满足：激活 `local` Profile、`app.sql-console.enabled=true`、调用者具备 `system:sql:execute`。其他 Profile 或关闭开关时服务返回不可用，不执行 SQL。

单次请求只允许一条 SQL，最大 20,000 字符；查询结果最多返回 100 行并将 `password`、`token`、`secret`、`hash` 等敏感列掩码。写入、DDL 和 DCL 必须携带显式危险操作确认标识。每次请求保存审计事件，但审计元数据仅记录语句类别、长度、结果数量和 SQL 摘要，不保存原始敏感参数。

## 数据库与验证

新增 GC 策略、运行记录和锁表；增加 Refresh Session 维护索引。SQLite 与 PostgreSQL 均通过 Liquibase 迁移验证。测试覆盖策略初始化、dry-run 无副作用、保留期边界、并发锁、内存状态清理、SQL 环境门禁、单语句约束、敏感列掩码与权限边界。
