# 后端安全、性能与可观测性加固设计

## 目标

在不改变既有业务协议的前提下，收紧本地 SQL 调试台的写入边界，消除 IP 规则热路径重复查询，为 GC 调度提供可观测状态，并给可能持续增长的管理列表加入统一分页边界。

## 设计

1. SQL 执行台采用保守分类：仅 `SELECT`、不含写入 CTE 的 `WITH`、普通 `EXPLAIN` 视为只读；其余语句都需要危险操作确认。所有写入在单条语句校验后执行并审计语句类别，不记录 SQL 原文。
2. IP 规则服务维护不可变规则快照。启动、创建、删除规则后刷新一次；过滤器对单请求只读取一次快照并完成 allow/deny 判定。保留现有运行时配置规则优先级。
3. GC 调度器持有内存状态：最近尝试、最近成功、最近失败、连续失败数与预计下次检查时间。调度失败写 WARN 日志；管理 API 返回状态，Actuator health 在连续失败时报告 `DOWN`。
4. 默认生产环境关闭 Springdoc；`local` 显式开启。关闭 JPA Open Session in View。公告管理列表使用分页和页大小上限；其他固有小型目录维持全量读取。

## 约束

- SQL 控制台仍必须同时满足 `local` Profile、`app.sql-console.enabled=true` 和 `system:sql:execute`。
- 不新增 Redis 或消息队列；IP 快照的作用域是单实例。
- 全部改动先写失败测试，再实现；SQLite 全量测试必须通过。
