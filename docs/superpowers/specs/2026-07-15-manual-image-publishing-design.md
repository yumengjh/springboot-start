# 手动镜像发布设计

## 目标

普通代码推送只验证项目质量，不再构建或发布容器镜像。需要发布镜像时，由维护者在 GitHub Actions 页面手动触发，并明确填写发布版本。

## 触发与版本

- `CI` 工作流继续在每次推送和 Pull Request 中运行。
- `Container image` 工作流只使用 `workflow_dispatch` 触发。
- 手动触发时必须填写 `version`，格式为稳定的语义化版本 `MAJOR.MINOR.PATCH`，例如 `0.1.0`。
- 触发页的输入说明提示当前 Maven 项目版本为 `0.0.1-SNAPSHOT`，帮助维护者选择下一正式版本。
- 工作流拒绝空版本、`SNAPSHOT` 版本、`v` 前缀和不符合三段数字格式的输入。

## 发布结果

一次成功发布只构建一次镜像，并在扫描通过后将同一归档推送到 GHCR。镜像标签为：

- 完整版本：`0.1.0`
- 次版本：`0.1`
- 默认稳定版：`latest`
- 可追溯构建：`sha-<短提交哈希>`

不再生成会随普通提交移动的 `main` 镜像标签。

## 安全与验证

- 继续对构建产物运行 Trivy；存在可修复的 CRITICAL 漏洞时阻止发布。
- 通过扫描后保留一天 Actions 归档，再由 publish job 加载并推送同一镜像，避免二次构建。
- 验证工作流 YAML、版本输入校验和手动发布标签规则；普通推送后确认不会出现镜像工作流。
